# Phase 4: Rust 移行の継続と重要バグの修正 仕様書

- **ブランチ**: `feature/phase4-rust-migration`
- **ステータス**: 計画中
- **作成日**: 2026-02-24
- **目標マージ日**: 2026-03-XX

---

## 1. 目的

Phase 3 で構築した Rust Bridge 基盤を最大限に活用し、残存する Python ロジックを Rust へ移行（疎結合化）を進めると共に、現在発生している「アンケート集計の不一致」および「自己修復機能の権限エラー」を解消し、システムの信頼性を向上させる。

## 2. 主要課題 (Core Pillars)

### 2.1 アンケート回答情報の取得失敗の解消
**事象**: ユーザーがアンケートに回答し、Discord で完了通知 DM を受け取っているにもかかわらず、管理ダッシュボード上の「回答数」が 0 件のまま表示される。
- **原因の推測**:
    - Rust Bridge 側の `get_survey_responses` 内部での JOIN 条件不一致。
    - 回答保存時（UPSERT）のデータ型変換ミス。
    - ダッシュボードクエリにおける論理削除フラグの誤判定。
- **対応方針**:
    - Rust Bridge のログを確認し、DB への保存状況と取得クエリの実行結果を突合する。
    - ユニットテストに「回答取得シナリオ」を追加。

### 2.2 自己修復機能（自動ミュート）の完全化
**事象**: Startup や Channel Update 時の自己修復プロセスにおいて、`Missing permissions to edit channel` エラーが発生し、一部チャンネル（`#mute_only`, `#readonly`等）の権限調整に失敗している。
- **原因の推測**:
    - Bot ロールが該当チャンネルの「上位」に位置していない。
    - カテゴリによる権限の強制上書き。
    - `discord.py` の `overwrite` 適用順序の不備。
- **対応方針**:
    - `PermissionService` のロジックを見直し、ヒエラルキーの検証ステップを追加。
    - エラー発生時の詳細な監査ログ（Audit Log）の出力と再試行制御の実装。
    - 将来的にはこの「権限判定エンジン」自体を Rust Bridge 側に持たせることを検討。

### 2.3 ロジックの Rust 移行 (Migration Roadmap)
**対象**:
- **権限判定ロジック**: 現在 Python 側にある「どのロールがどのチャンネルに対しどうあるべきか」の定義を Rust Bridge の Config/DB へ移動。
- **通知メッセージのテンプレート管理**: DM 送信内容の選定を Rust 側で行い、Python は API 経由で命令を受け取るだけに留める。
- **操作ログ (Audit) の完全集約**: Python 側の全ログを Rust Bridge 経由で保存し、DB アクセスを完全に隠蔽。

## 3. 実装フェーズ

### フェーズ 4.1: 緊急バグ修正 (Hotfix)
1. アンケート回答取得ロジックのデバッグと修正。
2. 自己修復機能の権限チェック機能の強化。

### フェーズ 4.2: 権限エンジンの移行
1. 権限設定データ（JSON/DB）の Rust 側への定義移行。
2. Rust 側で「期待される権限状態」を算出し、Python 側に差分を通知する仕組みの構築。

### フェーズ 4.3: ログと通知の統合
1. 通知サービスの一部 Rust 化。
2. 監査ログ機能の完全 Rust 化。

## 4. 成功基準 (Success Criteria)

1. **データ整合性**: アンケート回答がリアルタイム且つ正確にダッシュボードに反映されること。
2. **自己修復率 100%**: 権限エラー報告が 0 になり、定義通りの権限が維持されること。
3. **Python コード削減**: データベース操作を直接行う Python コードが 0 になり、API 経由でのアクセスのみになること。

## 5. バグ予備診断 (Preliminary Diagnosis)

### 5.1 アンケート回答が 0 件になる件
- **現況**: Discord DM は送信されており、Rust 側の `upsert_response` は正常終了（ID 返却）していると推測される。
- **疑わしい箇所**:
    1. **`response_repo::find_by_survey` のクエリ**: `survey_id` の型（signed/unsigned）や、DB 上の外部キー制約の有無。
    2. **`submitted_at` のデコード失敗**: `sqlx` が `OffsetDateTime` へのマッピングに失敗し、結果セットが空として扱われている（または暗黙的にエラーがスキップされている可能性）。
    3. **Rust Bridge の再起動漏れ**: Rust 側のコード変更が反映されていない古いバイナリが動いている可能性。
- **調査手順**:
    - `ssh` で直接 MariaDB に入り、`survey_responses` テーブルの中身を SELECT してデータ存在を確認。
    - Rust Bridge のログに `Sqlx error` が出ていないか確認。

### 5.2 自己修復機能の権限エラー (`Missing permissions`)
- **現況**: チャンネル個別の `overwrite` 適用時に発生。
- **主な原因**:
    - **ロール順序**: Bot 自身の最高ロールが、操作対象のロール（@everyone 等）より上位にある必要がある。
    - **チャンネル固有権限**: Bot のロールに「チャンネルの管理」または「権限の管理」が**そのチャンネル内で**明示的に許可されている必要がある。
- **改善案**:
    - `PermissionService` に「権限不足」を検知した際の管理者通知（DM）機能を追加。
    - 設定ファイルに基づき、Bot が本来持っているべき権限を事前にセルフチェックする機能を実装。

## 6. 結論

Phase 4 は単なる移行作業に留まらず、本番稼働で露呈した「データの透明性」と「自律制御（自己修復）の信頼性」を確固たるものにする重要なフェーズである。本仕様書に基づき、順次 `feature/phase4-rust-migration` ブランチにて実装を進める。

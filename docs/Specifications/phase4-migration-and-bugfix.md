## 1. 目的

Phase 3 で構築した Rust Bridge 基盤を最大限に活用し、残存する Python ロジックを Rust へ移行（疎結合化）を進めると共に、現在テスト環境で発生している「アンケート集計の不一致」および「自己修復機能の権限エラー」を解消し、システムの信頼性を向上させる。

## 2. 主要課題 (Core Pillars)

### 2.1 アンケート回答情報の取得失敗の解消 ✅ Phase 3-D で解消済み
**事象**: ダッシュボード上の「回答数」が 0 件のまま表示される不具合。
- **原因**: MariaDB と Rust(sqlx) 間の `DATETIME` / `LONGTEXT` の型不一致。
- **解決**: SQL `CAST` の導入と `Vec<u8>` 受け、UNIQUE KEY による Upsert 正常化。詳細は `phase3d-hotfix-survey-and-massmute.md` 参照。

### 2.2 自己修復機能（自動ミュート）の完全化 ✅ Phase 3-D で解消済み
**事象**: Startup や Channel Update 時の `Missing permissions` エラー。
- **原因**: チャンネル固有の権限上書きによる Bot のブロック。
- **解決**: サーバー全体の「ロールの管理」権限を用いた「セルフ・アンブロッキング」ロジックの導入。診断コマンド `!mute_check` の追加。
- **今後**: 権限判定ロジック自体の Rust 移行を Phase 4.2 で進める。

### 2.3 ロジックの Rust 移行 (Migration Roadmap)
**対象**:
- **権限判定ロジック**: 現在 Python 側にある「どのロールがどのチャンネルに対しどうあるべきか」の定義を Rust Bridge の Config/DB へ移動。
- **通知メッセージのテンプレート管理**: DM 送信内容の選定を Rust 側で行い、Python は API 経由で命令を受け取るだけに留める。
- **操作ログ (Audit) の完全集約**: Python 側の全ログを Rust Bridge 経由で保存し、DB アクセスを完全に隠蔽。

## 4. 成功基準 (Success Criteria)

1. **データ整合性**: アンケート回答がリアルタイム且つ正確にダッシュボードに反映されること。
2. **自己修復率 100%**: 権限エラー報告が 0 になり、定義通りの権限が維持されること。
3. **Python コード削減**: データベース操作を直接行う Python コードが 0 になり、API 経由でのアクセスのみになること。

## 6. 結論

Phase 4 は単なる移行作業に留まらず、テスト環境での稼働で露呈した「データの透明性」と「自律制御（自己修復）の信頼性」を確固たるものにする重要なフェーズである。本仕様書に基づき、順次 `feature/phase4-rust-migration` ブランチにて実装を進める。

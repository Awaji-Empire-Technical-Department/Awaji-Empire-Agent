# ADR 010: セキュア対戦ロビーシステムのWebSocket化と大会支援機能拡充 (Phase 4.2)

## ステータス

提案中 (Proposed) / 実装済 (Implemented)

## 背景と課題 (Context & Problem Statement)

セキュア対戦ロビーシステム（Phase 4.1）において基本的なロビー作成とIP交換機能は実現したが、以下の課題が残っていた。

1. **リアルタイム性の欠如**: これまでのシステムでは、ロビーの参加者増加やホストのステータス変更（受付中→対戦中への移行など）をユーザーが知るためには、ブラウザを都度リロードする必要があった。
2. **大会進行支援の自動化不足**: 大会（Tournament）モードでの、自動的な対戦テーブル（ブラケット）の生成、ホスト・クライアントの指定機能、勝敗報告、およびDiscord連携（優勝ロール付与）が未実装であった。
3. **VM環境の制約**: 当初予定していたフロントエンドのTypeScript+Vue/React化などのビルド環境構築は、既存の仮想マシン（VM）環境にNode.jsツールチェーン等の新規導入が必要となり、運用コストが高くなる懸念があった。

## 意思決定 (Decision)

上記の課題と運用上の制約を考慮し、以下のアーキテクチャおよび技術選定を行った。

### 1. リアルタイム通信 (WebSocket) の導入

Rustバックエンド（`database_bridge`）の `axum` フレームワークにおいて直接 WebSocket エンドポイント (`/ws/hyouibana`) を提供する。

* **ブロードキャスト処理**: `tokio::sync::broadcast` を用いたチャンネルを `AppState` に持たせ、ロビーに関する既存のREST API（参加、更新、削除等）が実行されたタイミングで、関連する全WebSocketクライアントへ JSON形式のイベント（`member_status_updated`, `match_created` 等）を送信する。
* これにより、ユーザーのブラウザリロード無しでの完全な状態同期を実現。

### 2. 大会進行とブラケット可視化ライブラリの選定

* **不採用案**: 複雑なフロントエンドフレームワークの導入。
* **採用案**: 既存のQuart/Jinja2テンプレートを維持したまま、モダンで外部依存のないJavaScriptライブラリ **`Bracketry`** を CDN 経由で導入する。フロントエンドのロジックはVanilla JavaScript (ES6 Modules: `possession_lobby.js`) で記述し、VMへの新たなビルド環境追加を回避する。

### 3. データモデルの拡張と Discord Bot 連携

* **バックエンド (Rust/DB)**: ロビー参加者に `status` (オンライン、受付中、待機中など)を付与し、さらに大会の対戦カードを追跡するため `tournament_matches` テーブルに `round_num`、`match_index`、`score` などのカラムを追加（マイグレーション `006_tournament_updates.sql`）。
* **フロントエンドからDiscordへの橋渡し**: ユーザー（ホスト）の「最終承認」アクションを受け、Python（Quart）サーバーから HTTPX を経由して Discord REST API (`/v10/guilds/{id}/roles`) を直接コール。指定された大会名の「〇〇優勝」ロールを動的に生成・付与する。これをバックグラウンドタスクや既存Bot側の常駐プロセスとは非同期かつ直接的に連携させた。

## 結果 (Consequences)

### ポジティブな影響

* **圧倒的なUX向上**: ユーザーはページ更新の手間なく、参加者の入室や対戦の開始状況をリアルタイムに把握可能となった。
* **運営コストのゼロ化**: 大会進行から優勝ロールの付与までが一気通貫で自動化されたため、大会主催者の管理コストが大幅に削減される。
* **シンプルな技術スタックの維持**: `Bracketry`とVanilla JSの採用により、ビルド構成を複雑化することなく高度なUIを実現できた。

### ネガティブな影響 / 留意点

* **WebSocketの運用管理**: Rustプロセスが長期間稼働しコネクションを保持し続けるため、タイムアウトやゾンビコネクションの検知・切断機構の監視が必要になる（現状は `axum` の標準タイムアウト等に依存）。
* **UI表示モックデータの置き換え**: 今回フロントエンドに実装したBracketryのレンダリング部には一部ダミー構造が含まれるため、後日Rustから渡管された `matches` API のレスポンスを直接マップするよう JS 側のデータ変換関数の本番実装が必要である。

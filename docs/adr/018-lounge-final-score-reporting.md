# ADR-018: ラウンジ申告方式変更 — セッション最終順位申告方式

- **ステータス**: 採用
- **作成日**: 2026-05-23
- **作成者**: Wanyaldee
- **関連ADR**: [ADR-013](013-lounge-system.md), [ADR-016](016-lounge-ux-realtime-improvements.md), [ADR-017](017-lounge-bugfix-session-result.md)

---

## コンテキスト

ラウンジシステムは当初、レースごとに全員が順位を申告し、ホストが承認するという方式で設計・実装された。
しかし、淡路帝国の参加型配信という運用特性において、レースごとの申告操作がゲームの流れを断絶させるという問題が顕在化した。

---

## 決定内容

**レースごとの順位申告フローを廃止し、セッション終了後に最終順位を1回だけ申告する方式に変更する。**

### 変更前（旧方式）
1. ホストがレース作成（コース名入力）
2. 各プレイヤーがそのレースの順位を申告
3. ホストが承認 → 配点テーブルでポイント計算
4. 12レース繰り返し → セッション終了

### 変更後（新方式）
1. 12レースをゲーム内でプレイ
2. セッション終了時に全員が最終順位（1〜24位）を1回申告
3. ホストが終了確定 → MMRデルタテーブルで一括計算
4. MMR・称号を結果モーダルに表示

---

## 理由

- **配信体験の優先**: 参加型配信では視聴者がゲームの流れを見ている。レースごとの中断は視聴体験を損なう
- **実装のシンプル化**: per-race の申告・承認フローより最終順位1回の方が実装コストが低く、バグが出にくい
- **配点ロジックの単純化**: 累積ポイント計算を廃止し、MMRデルタテーブルに一本化することで理解しやすくなる

## 廃止した設計要素

| 廃止物 | 理由 |
|-------|------|
| `lounge_race_results` テーブル | コース管理不要になったため |
| `lounge_race_scores` テーブル | per-race スコア管理廃止 |
| `lounge_course_history` テーブル | コース重複検知廃止 |
| `point_tables`（ラウンジ用）参照 | MMRデルタテーブルに置換 |
| `report_score` / `approve_race_scores` 等 API | 申告フロー廃止 |
| `#modal-phase-report` / `#modal-phase-setup` | UI全面置換 |

---

## 追加した設計要素

| 追加物 | 用途 |
|-------|------|
| `lounge_session_final_scores` テーブル | セッション最終順位申告を保持 |
| `lounge_session_members.excluded` カラム | ホストによる除外フラグ |
| `POST /lounge/sessions/{id}/final-scores/report` | 最終順位申告 API |
| `POST /lounge/sessions/{id}/exclude` | 除外トグル API |
| `lounge.final_score_reported` WS イベント | 申告状況のリアルタイム共有 |
| `lounge.member_excluded` WS イベント | 除外操作のリアルタイム共有 |

## トレードオフ

| 観点 | 旧方式 | 新方式 |
|-----|-------|-------|
| 配信体験 | 中断あり | 中断なし |
| 不正申告リスク | レースごとに検証可能 | 最終順位のみで検証困難（スクリーンショット担保） |
| 途中経過の見える化 | リアルタイムランキング有 | セッション中はランキングなし |
| 実装複雑度 | 高い | 低い |

不正申告の検証については、既存の「スクリーンショット必須」警告（UI上に常時表示）で運用上カバーする。

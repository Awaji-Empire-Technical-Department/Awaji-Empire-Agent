# ADR 014: 配信中リセットガード（フェールセーフ）

## ステータス

実装済・更新 (Updated)

## 背景

`/reset_stream_comments` スラッシュコマンドを配信中に誤発火させてしまった際、チャンネル削除→再作成によって Discord がリスナー全員に通知を送り、その通知音が作業等を妨害するという問題が発生した。

通知の無効化を試みたが Discord の仕様上、チャンネル再作成通知はユーザー側の設定に依存するため、Bot 側から完全に回避する手段がなかった。

## 決定（最新）

`/reset_stream_comments`（`dry_run=False`）実行時に、`TARGET_USER_ID` が VC に在席中であれば即座にブロックするのではなく、`_pending_reset` フラグを立てて予約し、ホストが VC を退室した時点（`on_voice_state_update`）で自動実行する。

これは yuyq の当初意図（「配信中に打ったコマンドを終了後に自動で実行したい」）に沿った設計変更である。当初のブロック方式は実装コストと心理的摩擦の観点から採用したが、「配信が盛り上がって削除し忘れる」という実運用上の問題が明らかになったため変更した。

### 変更前の決定（初版）

`/reset_stream_comments`（`dry_run=False`）実行時に、`TARGET_USER_ID` が VC に在席中であれば即座にブロックする。

- **配信中の定義**: `TARGET_USER_ID`（社畜天狗）が任意の VC チャンネルに在席している状態
- **配信終了の定義**: Voice Keeper が寝落ち集計報告を投稿した時点（= `is_host_in_vc` が `False` になる）
- `dry_run=True` は破壊的操作を伴わないため、配信中でも許可する
- 自動トリガー（VoiceKeeper 報告・fallback cron）は設計上すでに配信終了後にのみ発火するため対象外

## 実装（最新）

- `logic.py`: `StreamCommentResetLogic.is_host_in_vc(guild, target_user_id) -> bool` を追加
- `cog.py`:
  - `_pending_reset: bool` フラグを追加
  - `reset_stream_comments` コマンドで配信中の場合は `_pending_reset = True` にして予約メッセージを返す
  - `on_voice_state_update` リスナーを追加: ホスト退室時に `_pending_reset` が True なら `_try_monthly_reset(force=True)` を実行
  - `_try_monthly_reset` に `force: bool = False` 引数を追加、`force=True` のとき冪等チェックをスキップ

## 却下した選択肢

**A. 配信フラグをメモリ上で管理する**
- `on_voice_state_update` で `is_streaming` フラグを更新する案
- ホストの VC 入室は VoiceKeeper が監視していないため追加実装が必要
- Bot 再起動でフラグが消失するリスクがある
- リアルタイムの `member.voice` チェックで十分なため不採用

**B. 配信中の通知を完全に抑制する**
- Discord API にはチャンネル再作成通知を Bot 側から抑止する手段がない
- 根本解決にならないため不採用

## トレードオフ

- ホストが VC に居る間は管理者による緊急リセットも不可能になる
- これは意図的な制約（配信中は通知音の方が問題）
- 緊急時は `dry_run=True` でプレビューを確認し、配信終了後に実行する運用で対応

## 却下した選択肢への補足：「キュー方式（配信中にコマンドを受け付け、終了後に自動実行）」を採用しなかった理由

配信中にコマンドを受け取り、ホストが VC を退室した時点で自動的にリセットを実行するキュー方式は一見合理的に見える。しかし以下の理由で不採用とした。

**実装コストの問題:**
- `on_voice_state_update` によるキュー消費ロジック、キューの永続化（Bot 再起動耐性）、二重実行防止など、コード量が大幅に増加する。

**心理的意図（設計上の意図的な摩擦）:**
- 単純にブロックすることで、「必要があるなら配信終了後に改めてコマンドを打つだろう」という運用上のクールダウンを促す。
- 配信中は興奮・混乱状態での誤操作リスクが高い。一度ブロックすることで、操作者が「本当に実行すべきか」を落ち着いて判断するタイミングを強制的に作る設計。
- キュー方式では「配信中に打ったコマンドがいつ実行されるか分からない」という新たな認知負荷が生まれる。

**結論（初版時点）:** シンプルなブロックの方が実装・運用の両面で優れていると判断した。

**→ 後日変更:** 実運用で「配信が盛り上がって削除し忘れる」ケースが問題視され、キュー方式（`_pending_reset` フラグ）として部分的に採用した。ただし「全キュー永続化」は不採用のまま（Bot 再起動でフラグ消失は許容）。

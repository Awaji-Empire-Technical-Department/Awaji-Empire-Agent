# ADR-021: イベント参加フォームシステム

- **ステータス**: 採用
- **作成日**: 2026-05-24
- **作成者**: Wanyaldee
- **関連ドキュメント**: [仕様書](../event_form_spec.md), [ADR-005](005-phase3d-survey-response-fix.md)

---

## コンテキスト

Awaji Empire では年に一度程度のオフ会を開催している。2023年の開催時は複数の「部」に分けた時間帯制を採用した。これまでは参加募集・参加者管理・個別通知を手作業で行っており、主催者の負担が大きかった。

既存のアンケート機能（survey）は汎用的な質問収集に対応しているが、以下の点でイベント管理には不足している：

- 部（時間帯）ごとの定員管理ができない
- 参加可否を個人ごとに伝える手段がない
- 日程・場所・参加費の情報を自動で通知できない

---

## 決定内容

既存のアンケート機能を**拡張**する形でイベント参加フォームを実装する。アンケート作成画面に「イベント参加フォームとして作成する」チェックボックスを追加し、有効化した場合のみイベント管理機能が有効になる設計とした。

### データ設計

3テーブルを新設する。

```
events           — アンケートに1:1で紐づくイベントのメタ情報
event_sessions   — 部（0件なら部制なし）
event_participants — 応募者ごとの割り当て・承認状況
```

`event_sessions` が0件の場合を「部制なし」として扱い、専用フラグは持たない。シンプルなデータで部制の有無を表現できる。

### 部制と日時

- **部制あり**: 各 `event_session` が `event_date` / `end_date` を持つ。参加者は割り当てられた部の日時でカレンダー登録する。
- **部制なし**: `events` テーブルの `event_date` / `end_date` を使用。全員共通の日時でカレンダー登録する。
- `end_date` が未設定の場合は `event_date + 2時間` をデフォルトとしてカレンダーURLを生成する。

### 自動割り当てロジック

Rust側 (`db/event_repo.rs`) に実装し、以下の優先順位で割り当てる：

```
1. 不参加（希望なし）→ スキップ
2. 第一希望の部に空きあり → accepted
3. 第一希望が満席 → 空きのある別の部へ振り分け
4. 全部満席 → waitlist（補欠）
```

自動割り当て後もオーナーが手動で上書きできる。

### 通知・確認

- **Discord DM**: Bot経由でオーナーが一斉送信。承認・否認・補欠で文面を分岐。
- **カレンダーリンク**: OAuthなし・URLパラメータのみで Google Calendar / Outlook の登録URLを生成し、承認DMと個人確認ページの両方に掲載する。
- **個人確認ページ** (`/event/confirm/<token>`): ランダムトークンによりログイン不要でアクセス可能。参加者がいつでも自分の割り当て情報・カレンダーリンクを確認できる。

### 応募要件

イベントフォームへの応募にはDiscordログインが必須。応募者の特定・DM通知にDiscord user_idを使用するため、未ログイン時はフォームの送信を非活性にする。

## 検討した代替案

### 案A: 専用の募集フォームを別途作成する

アンケートとは完全に独立したイベント申込フォームを実装する案。

- **却下理由**: 質問の柔軟な追加・条件分岐（Branch Logic）など、既存のアンケートエディタに同等の機能を再実装するコストが高い。既存資産を活かす方が合理的。

### 案B: Google Forms / Tally.so などの外部サービスを使う

- **却下理由**: 回答データが外部に出るため、Discord user_idとの紐づけが困難。個別通知のオートメーションも外部サービスに依存することになりシステム統合性が失われる。

## 将来的な拡張候補（未実装）

### 参加費の事前オンライン決済（Stripe）

現在、参加費は「当日現金払い」を前提としており、システムでの支払い管理は行っていない。

将来的に事前決済を導入する場合は **Stripe** の採用を推奨する。

**理由:**

- Stripe は決済ごとの明細・領収書を自動発行するため、**確定申告時の証憑管理が容易**になる。
- Stripe Dashboard 上でイベントごとの入金状況を一元管理できる。
- `event_participants` に `stripe_payment_intent_id` カラムを追加し、支払い完了フックで `approval` を `accepted` に自動遷移させる設計が自然に組める。
- 個人間での現金授受が減ることで、主催者・参加者双方の金銭トラブルリスクを低減できる。

導入時は収益が発生するため、**法人格の有無・確定申告の要否**を事前に確認すること。

参考
- [Stripe 公式ドキュメント](https://stripe.com/docs)
- [【売上の取りこぼしを防げ】利益率を左右するのは「決済」だった／決済成功率90％／売上の機会損失をゼロにする「次世代金融インフラ」の全貌／LGがStripeを選んだ理由は](https://www.youtube.com/watch?v=irldrMeA5FM)

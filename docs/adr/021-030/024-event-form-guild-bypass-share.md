# ADR-024: イベントフォーム回答側のギルドチェック緩和・回答リンク共有ボタン

- **ステータス**: 採用
- **作成日**: 2026-06-14
- **作成者**: Wanyaldee

---

## コンテキスト

イベント参加フォーム（ADR-021）は、Discord OAuth ログインを必須とし、ログイン時の `/callback` で
対象ギルド（Discord サーバー）への加入チェックを行っている。未加入ユーザーは `access_denied.html`
を返され、ダッシュボードの一切にアクセスできない。

```python
# discord_bot/webapp.py /callback（旧実装）
if Config.TARGET_GUILD_ID:
    # ギルド未加入なら 403 access_denied
    if str(Config.TARGET_GUILD_ID) not in guild_ids:
        return await render_template('access_denied.html'), 403
```

このフォームをオフ会の参加集計に使う運用を想定したとき、以下の懸念が生じた。

1. **取りこぼしリスク**: オフ会参加者は概ねギルド加入済みだが、最悪のケースとして
   「まだギルドに入っていないが参加したい人」の回答を 403 で弾いてしまう。
   集計用途では「全員分を確実に拾う」ことが最優先。
2. **共有手段の不足**: 回答ページ（`/form/<id>`）の URL を他者に渡す手段がフォーム上になく、
   主催者が手動で URL をコピーして配る必要があった。

一方で、ギルドチェックは管理系画面（フォーム作成・編集・結果閲覧・イベント管理）の
アクセス制御として依然必要であり、全面的に撤廃はできない。

---

## 決定内容

### 1. ギルドチェックを「回答フローのみ」スキップ

`/callback` のギルドチェックを、ログイン後の遷移先が回答ページ（`/form/`）の場合に限り
スキップする。遷移先は既存の `session['next_url']` 機構（`view_form` が未ログイン時に
`request.url` を保存）をそのまま利用するため、追加の状態管理は不要。

```python
# discord_bot/webapp.py /callback（新実装）
next_url = session.get('next_url', '') or ''
is_form_answer = '/form/' in next_url
if Config.TARGET_GUILD_ID and not is_form_answer:
    # 管理系画面は従来通りギルド加入を必須とする
    ...
```

| フロー | ギルド加入 | 根拠 |
|---|---|---|
| 回答（`/form/<id>`） | **不要** | オフ会集計で未加入参加者を取りこぼさない救済措置 |
| 管理系（作成・編集・結果・イベント管理） | 必須 | アクセス制御として維持 |

**Discord ログイン自体は引き続き必須。** 応募者の特定と DM 通知に `user_id` が必要なため
（ADR-021 / 仕様書 §3.2）、`view_form` / `submit_response` のログインチェックは残す。
今回の緩和は「ギルド未加入の Discord アカウントでもログインして回答できる」ことを意味し、
完全匿名（ログイン不要）回答を許可するものではない。

### 2. 回答リンク共有ボタンの設置

`form.html` のタイトルカードに「回答リンクを共有」ボタンを追加。フォールバック順は
モバイルのネイティブ共有を優先する。

```
navigator.share（Web Share API）
  → navigator.clipboard.writeText（クリップボードコピー）
  → window.prompt（手動コピー）
```

---

## DM の位置づけ（控え）

回答送信後、Bot が応募者へ送る DM（`submit_response` → `NotificationService.send_dm_raw`）は、
**応募内容の「控え」** としての意味を持つ。確認 URL（`/event/confirm/<token>`）を含み、
応募者が後から自分の応募内容を再確認できる。ギルド未加入者であっても DM 送信先は
`user_id` で解決できるため、控えの送付は問題なく機能する。

---

## 検討した代替案

### ギルドチェックを全面撤廃

- **却下理由**: 管理系画面（フォーム作成・結果閲覧・イベント管理）のアクセス制御が
  失われ、ギルド外の第三者が管理 UI に到達しうる。回答フローに限定すべき。

### 完全匿名（ログイン不要）回答を許可

- **却下理由**: 応募者の特定・DM 控え送付・確認ページのトークン発行が `user_id` に
  依存しており（ADR-021）、匿名化はイベント機能の根幹を崩す。ログインは維持する。

### フォーム側に専用の「ゲスト用 URL」を別途発行

- **却下理由**: URL 体系・トークン管理が増え、KISS 原則に反する。既存の `next_url`
  機構で回答フローを判別できるため、最小実装で目的を達成できる。

## 影響範囲

| 対象 | 変更内容 |

| `discord_bot/webapp.py` | `/callback` のギルドチェックを回答フロー（`/form/`）のみスキップ |
